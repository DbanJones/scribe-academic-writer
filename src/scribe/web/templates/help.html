How to use Scribe

A practical guide to the four modes, the conventions Scribe enforces, and the output you'll get back.

What is Scribe?

Scribe is a writing tool for academic prose. It takes one of three starting points -- an outline, an existing draft, or a finished paper -- and produces polished output against a unified set of academic writing rules (Allwood's Cambridge six-element structure, Williams on characters and actions, Pinker on classic style, Sword on zombie nouns, Schimel's hourglass, Zinsser on clutter, Amazon's weasel-word discipline, Graff & Birkenstein on citation as dialogue).

Everything Scribe writes is driven by Claude via the local Claude Code CLI. Your documents never leave your machine; usage is billed to your own Anthropic account or Max subscription.

Connect Scribe to your Claude account

Scribe does not talk to Anthropic's servers directly. Every API call goes through the Claude Code CLI installed on your machine. That means:

  • Your documents never leave your computer.
  • Usage is billed to your own Anthropic account (or your Max subscription) — not to Scribe or anyone else.
  • Scribe can't see your credentials; they live in the CLI's own settings (~/.claude/) or your OS keychain.

There are three pieces to get right: install the CLI, log in, verify it works. A healthy setup shows a green Connected chip in the top-right of every Scribe page.

Step 1 — Install the Claude Code CLI

If you already have the CLI or the Claude desktop app, skip to Step 2 — Scribe auto-detects either one.

macOS

Easiest route is Homebrew: 

brew install claude

Or download the installer from docs.claude.com.

Windows

Install the Claude desktop app from claude.ai/download — it bundles the CLI. Scribe looks for claude.exe in the desktop app's app-data folder automatically.

Or install via npm if you prefer a CLI-only setup: 

npm install -g @anthropic-ai/claude-code

Linux

Install via npm (Node.js required): 

npm install -g @anthropic-ai/claude-code

Verify with which claude; it should resolve somewhere on your $PATH.

Verify the install

Open a terminal and run: 

claude --version

If you see a version number, you're done with step 1. If you see "command not found", check your PATH or use the desktop app on Windows.

Step 2 — Log in

The first time you use Scribe, Claude Code needs to be logged into an Anthropic account. You have two equivalent options.

A
From Scribe's web UI (easiest). Click the amber "Login required" chip in the top-right nav, or open /login directly. Press "Login with Claude". The CLI opens your default browser so you can sign in. Once you approve, the page will flip to "Connected" and you can close that tab.
B
From a terminal. Run claude login and follow the browser prompt. Scribe picks up the session automatically the next time you refresh.

Either path asks you to sign in with the Anthropic account that owns the API credits or Max subscription you want to bill. If you have multiple accounts, the CLI remembers whichever you chose last; run claude logout and log in again to switch.

Max subscribers: no extra configuration. Scribe uses whatever entitlements the CLI has. A Max subscription covers Opus and Sonnet usage, which is what every Scribe pipeline calls under the hood.

Step 3 — Verify the connection

The top-right of every Scribe page shows one of three chips:

ChipWhat it meansWhat to do
Connected Claude CLI installed and logged in. Pipelines will run. Nothing. You're good.
Login required CLI installed but not logged in (or your session expired). Click the chip → press "Login with Claude" → sign in.
Install CLI No claude binary found on your PATH or in the desktop-app location. Go back to Step 1.

Hover over the chip to see the exact path Scribe is using for the CLI. If that path looks wrong (points at an old version, say), uninstall the stale copy so Scribe picks up the current one.

What Scribe does with your connection

For every run, Scribe spawns Claude Code as a subprocess with a constrained tool set and your project folder as the working directory. The model has read-only access to the files you've uploaded plus the ability to use web search when the outline tag [web] is set. Scribe never passes credentials across the Python/SDK boundary; authentication happens entirely inside the CLI process.

Every pipeline stage is a short, self-contained call. If you see rate-limit warnings, lower parallelism in config.yml (default is 3 concurrent sections) or wait a few minutes before retrying.

Common connection problems

"Claude Code not found at <path>"

The CLI was installed but the binary isn't on your PATH where Scribe looks. Fixes in order of effort:

  • macOS/Linux: run which claude; if it resolves, restart the web server (scribe web) so it picks up the current PATH.
  • Windows: install the Claude desktop app if you haven't — Scribe looks for it at %LOCALAPPDATA%\packages\Claude_*\...\claude.exe automatically.
  • If you installed via npm, check that your npm global bin is on your PATH (npm config get prefix).
The login page opens a browser but never comes back to "Connected"

The CLI's OAuth flow runs in a different process from the web UI. Give it up to 30 seconds after you approve. If it still shows "Login required":

  • Reload /login to re-check status.
  • Open a terminal and run claude login directly; some auth errors only show up there.
  • Check the "Activity" panel on the /login page for the CLI's own stderr — it usually explains what went wrong.
"Session expired" or sudden auth failures mid-run

Long-running pipelines occasionally hit a session timeout or rate-limit. Scribe saves per-section output as it goes, so:

  • Go to /login and re-authenticate.
  • Resume the run: for Write mode use scribe resume <project>; for Revise and Expand, just re-submit the job — the cached audit / plan is reused and only the missing sections re-run.
Is my Anthropic API key stored anywhere in Scribe?

No. Scribe never reads or writes API keys. The Claude Code CLI handles all credentials in its own config (~/.claude/ on macOS/Linux, the OS keychain on some setups). Scribe only launches the CLI as a subprocess and listens to its stdout.

I want to switch between multiple Anthropic accounts

The CLI stores one active session at a time. In a terminal: 

claude logout
claude login

The second command asks you to sign into whichever account you want. Scribe will immediately reflect the new session the next time you hit /login or refresh.

First-time setup

  1. Connect to Claude. See the Connect to Claude section above — install the CLI, log in, confirm the green "Connected" chip at the top-right of every Scribe page.
  2. Pick a mode from the Dashboard and feed it the right input. See Which mode should I use? below if you're not sure which one fits your situation.
  3. Watch the progress track. Every mode shows a 4-stage progress bar with live log; runs typically take 5–25 minutes depending on length and mode. Scribe saves per-section output as it goes, so if anything fails partway through you can resume or re-submit and only the missing sections re-run.
  4. Download the results as either formatted Word (.docx) or Markdown (.md). Both are written to your chosen output folder; the web UI also offers one-click downloads from the mode pages.

Which mode should I use?

I have an outline, no prose

Headings and bullets, maybe a few references. You need Scribe to produce the first draft.

I have a finished draft, right length

The substance is there. The prose needs to be tighter, more concrete, more academic. Word count stays similar.

Revise →

I have a draft, need it longer

A summary or short draft (1,500+ words) to expand into a full chapter or journal submission. Double or triple the length with added depth.

Expand →

The paper is done but the target journal wants different formatting (Harvard to IEEE, APA to Nature, etc.).

Important: Scribe enforces "right input for the right mode". If you upload a finished 6,000-word draft as the outline in Write mode, it will refuse and suggest Revise or Expand. The modes are not interchangeable.
Write

Write a new document from an outline

Outline + references → new prose generated section by section against your style.

What you need

  • outline.md — headings and bullets only. A scaffold like "# Introduction / - Opening context / - Thesis statement" is not enough; bullets must be specific to your topic.
  • refs/ — optional PDF, DOCX, XLSX, MD, TXT, or CSV source material. The planner decides which references belong to each chunk.
  • style.md — optional custom style guide. The built-in academic rules apply regardless.

How Scribe runs it

  1. Document Review (Opus) — reads everything and identifies your problem statement, research gap, key question, themes, and how each section contributes to the argument.
  2. Plan (Opus) — chunks the document, assigns word budgets and references per chunk, and flags missing sources.
  3. Review TUI — approve or edit the plan before anything is written (web UI defaults to auto-approve).
  4. Write (Sonnet, in parallel) — each chunk is written with its thesis role injected.
  5. Stitch (Opus) — smooths transitions and enforces terminology consistency.

Walkthrough

1
Prepare an outline.md. A useful outline looks like this: 
# Chapter 3: Energy demand modelling

## Introduction
- Framing of the problem
- Why existing models fail

## ADIEM Layer 4 methodology [depth:rigorous]
- Tile-level energy demand estimation [ref:refs/adiem_notes.md]
- Monte Carlo uncertainty treatment
- Calibration against Ookla data [ref:refs/ookla_sample.xlsx]

## Results [depth:deep]
- Country-level outputs
- Sensitivity analysis
2
Open New project. Pick an output folder, drop the outline into the dropzone, upload any supporting PDFs or DOCX files, and click Create project.
3
On the project page, click Run Pipeline. The 4-stage progress track shows Plan → Write → Stitch progress; a 10,000-word chapter takes roughly 12–20 minutes.
4
When it finishes, download the result as Word (.docx) or Markdown. The document review (problem/gap/question/themes) is shown on the page.
Revise

Sharpen an existing draft

Existing prose → the same prose, only tighter. Length stays roughly the same. Citations, numbers, and argument preserved exactly.

What you need

  • A substantive draft (1,500+ words) with headings and citations. .md, .docx, .doc, or .txt.
  • Optional custom style guide.

What gets preserved

Preserved exactlySharpened
Every citation (inline and in the reference list) Zombie nouns → concrete action verbs
Every number, percentage, range, unit, date Clutter phrases → concise alternatives
Every figure and table reference Weasel words and unwarranted certainty
Your argument, framework, and terminology Sentence rhythm and stress positions
Section structure and heading hierarchy Transitions between sections (light smoothing)

How Scribe runs it

  1. Parse — split into heading-delimited sections.
  2. Audit (Opus) — produces a structured audit.json with per-section issues (category, severity, quoted text, suggested fix), plus document-level assessment of the hourglass shape and six elements.
  3. Revise (Sonnet, parallel) — each section is revised with its audit findings as targeted guidance, under hard constraints on preservation.
  4. Smooth (Opus) — a light consistency pass across all sections.

Tested example

On the test document ("So What?" ICT energy literature summary), 6,367 words → 5,635 words in 11m04s. 14/14 citations preserved, 0 hallucinated, structural bug (misplaced content between Gap 3 and Gap 4) detected and fixed. Audit report lists every specific change the reviser was asked to make.

Expand

Deepen an existing draft

Existing prose → longer prose with added depth. Same thesis, same citations, same claims — just unpacked further.

What you need

  • A substantive draft (1,500+ words) with headings and citations.
  • A target length: either a multiplier (e.g. 2.0x) or an exact word count.
  • Optional refs/ — source material Scribe can read for deeper context and (sparingly) cite from. No sources are ever invented.

What Expand will and won't add

Added depthNever introduced
Unpacking compressed claims into mechanismsNew substantive claims the author didn't make
Surfacing evidence that was impliedNew citations unless from uploaded refs
Addressing counter-arguments that belong in scopeContradictions of existing statements
Definitions of terms currently assumedDrift from the author's thesis
Methodology spelt out where skimmedRounded or altered numerical data

How Scribe runs it

  1. Parse — split into sections.
  2. Plan (Opus) — produces an expansion plan with per-section word budgets, "preserve" notes, and 2–6 specific expansion targets per section (e.g. "mechanism_compressed", "thin_evidence", "counterargument_missing").
  3. Expand (Sonnet, parallel) — each section gets its plan, the preserved-framework list, and available refs. May use Read/Grep to consult refs but cannot invent citations.
  4. Smooth (Opus) — final consistency pass.

Tested example

Same test document (6,244 words → 15,000 target, 2.4x) produced a plan with 47 expansion targets across 17 sections. Preserved framework correctly identified the author's coined terminology (stabilisation vs explosion camps, five Gaps, PRE metric, subsector carbon-intensity formula) as non-negotiable.

Restyle

Change a paper's citation style

Finished paper → same content, different citation format. Prose is unchanged except where rewording is strictly needed.

Supported styles

Harvard, APA, IEEE, Vancouver, Chicago author-date, Nature, Science, Journal of Cleaner Production, Journal of Industrial Ecology, Applied Energy, Energy Policy, IEEE Proceedings.

You can also build a custom preset form-driven from the Restyle page (live preview, optional free-text rules). Saved presets live in src/scribe/citations/styles/custom/ and are reusable.

Walkthrough

1
Open Restyle. Pick a source document, either by path or upload.
2
Choose a base style from the dropdown. The preview panel shows a before/after on a sample citation.
3
Optionally add free-text rules (e.g. "abbreviate months; DOIs on separate line") and save as a preset.
4
Click Restyle. Download the output as .docx or .md.

Outline tags (Write mode)

When writing a new document from an outline, drop these inline tags on headings or bullets to control generation per section.

TagWhat it doesExample
[depth:skim] Quickest treatment; minimum words Appendix context sections
[depth:standard] Default depth Most body sections
[depth:deep] Extra detail, more references pulled in Core analysis sections
[depth:rigorous] Maximum unpacking; longest treatment Methodology, critical findings
[ref:path/to/file.pdf] Force a specific reference into this chunk On a bullet that has a known source
[ref:path#pages-5-10] Same as above, limit to specific pages Long PDFs where only one section matters
[web] Allow web search for recent facts Sections about current events or markets
[skip] Exclude from generation entirely Notes-to-self you don't want rendered
[words:N] Force a specific word count Sections with a strict journal limit

Writing rules Scribe applies

Every generation, revision, and expansion call carries the same unified academic-writing rules preamble. It draws on:

  • Allwood — Cambridge six-element paper structure (Context, Literature, Proposal, Test design, Results, Discussion) and the hourglass shape.
  • Williams & Bizup — characters as subjects, actions as verbs, old-to-new information flow.
  • Pinker — classic style, syntactic tree management, curse-of-knowledge awareness.
  • Sword — concrete nouns, vivid verbs, no zombie nouns.
  • Schimel — story arcs at every level, stress-position emphasis.
  • Strunk & White — omit needless words, parallel structure, positive form.
  • Zinsser — clutter as disease; replace wordy phrases with concise alternatives.
  • Graff & Birkenstein — citation as dialogue, not as footnote; position your contribution.
  • Amazon six-pager discipline — quantify magnitude, avoid weasel words ("clearly", "interestingly").
  • Moran — sentence rhythm, earn long sentences with short ones.

The full 43-rule synthesis ships with Scribe. You can read it at src/scribe/resources/academic_writing_rules.md or override by supplying a custom style guide.

Output formats

Every pipeline produces both a markdown file and a formatted Word (.docx) file. The .docx handles heading hierarchy, bullets, numbered lists, tables, bold/italic/code, hyperlinks, figure captions, boxed visual suggestion placeholders, and bibliography entries with hanging indent.

ModeMarkdownWord
Write <project>/final.md <project>/final.docx
Revise <project>/revised.md <project>/revised.docx
Expand <project>/expanded.md <project>/expanded.docx
Restyle same folder, .restyled-<style>.md same folder, .restyled-<style>.docx

Audit/plan artifacts (for Revise and Expand) live under <project>/.scribe/ and include both JSON (machine readable) and Markdown (human readable) formats.

Share Scribe with other users

Scribe runs locally on each user's machine, so "sharing" means getting the code installed on their computer. There are three routes, from most polished to most developer-friendly.

Shared requirement: every user still needs the Claude Code CLI installed with their own Anthropic account logged in. Scribe never runs anything against a shared server; each user pays for their own Claude usage.

A. Windows .exe (no Python knowledge)

Send them a zipped folder containing Scribe.exe plus bundled Python runtime. They unzip, double-click, browser opens.

  • ~300 MB unzipped, ~100 MB zipped
  • Will trigger Windows SmartScreen unless signed
  • Rebuild and resend the zip when you update

B. pipx install (technical users)

Your colleague runs one command; they get the scribe and scribe-desktop commands on their PATH. Updates via pipx upgrade scribe.

  • ~30 MB install, cross-platform
  • Requires Python 3.11+ and pipx
  • Works everywhere (macOS, Windows, Linux)

C. Clone the repo (contributors)

For people who will customise Scribe. They get the source tree, can change prompts, add citation styles, and so on.

  • Requires git + Python toolchain
  • Best for anyone who will edit Scribe itself

A — Build a Windows .exe

Build the packaged executable from the repo. Run once on Windows: 

pip install pyinstaller
python scripts/build_exe.py              # onedir build (recommended)
python scripts/build_exe.py --onefile    # single-file exe, slower launch

Output goes to dist/Scribe/ (a folder containing Scribe.exe and its runtime). Zip the whole folder and hand it to your colleague. They:

  1. Install the Claude Code CLI once (Claude desktop app from claude.ai/download is the easiest route on Windows).
  2. Unzip your folder anywhere (Documents, Desktop, etc.).
  3. Double-click Scribe.exe. A console window opens, the Flask server starts on a free port, and the browser opens to Scribe's dashboard automatically.
  4. Close the console window (or press Ctrl+C) to shut down.
Windows SmartScreen: an unsigned exe may show "Windows protected your PC" on first launch. Recipients click More info → Run anyway. For a one-click experience on a wider audience, sign the exe with an Authenticode code-signing certificate (they're ~£100/year from providers like Sectigo or DigiCert).

B — pipx install from GitHub

For colleagues comfortable with a terminal, this is lighter and easier to keep up to date. Prerequisites: Python 3.11+ and pipx. Then: 

pipx install git+https://github.com/DbanJones/scribe-academic-writer.git

scribe-desktop      # launches the web UI in the browser
scribe --help       # CLI equivalent

To update when you push changes: 

pipx upgrade scribe

C — Clone the repo

For anyone who will edit the prompts, add citation styles, or run tests: 

git clone https://github.com/DbanJones/scribe-academic-writer.git
cd scribe-academic-writer
pip install -e .
scribe web

Sharing caveats

Keep in mind: each user runs their own copy of Scribe against their own Claude account. There is no central server, no shared queue, no centralised billing. Projects live on each user's disk.

If you want multiple people to collaborate on the same document, the cleanest pattern is one of you running Scribe, the other reviewing the markdown or Word output (e.g. via git, OneDrive, or email). Scribe itself is a single-user tool.

Common issues

"outline.md is still the default scaffold shipped by scribe init"

The generator refuses to run if the outline is still the placeholder "# My Document / ## Introduction / ## Body / ## Conclusion" template. Writing from an empty scaffold produces a meta-document about writing, not a paper about your topic.

Fix: replace outline.md with a real outline, or if you actually have a finished draft, use Revise or Expand instead.

"Does not look like a substantive draft"

Revise and Expand reject inputs that are too short or lack structure. They expect 1,500+ words with headings and some citations. If you have only an outline or a short sketch, use Write mode instead.

"Claude CLI not found" / "Login required"

See the Connect to Claude section for step-by-step install and login instructions for macOS, Windows, and Linux. The short version: install the CLI (or the Claude desktop app on Windows), then either click the auth chip in Scribe's top-right nav or run claude login in a terminal.

Pipeline crashed partway through; what happened to my work?

Scribe saves per-section output incrementally:

  • Write: drafts at .scribe/drafts/, state at .scribe/state.json. Run scribe resume <dir> to pick up from the last completed chunk.
  • Revise: per-section revisions at .scribe/revisions/. Audit cached at .scribe/audit.json. Re-running reuses the audit and only re-runs the revise + smooth stages.
  • Expand: per-section expansions at .scribe/expansions/. Plan cached at .scribe/expansion_plan.json. Same reuse logic.
The Word export doesn't look right

Open the markdown file and spot-check: Scribe's exporter relies on clean markdown structure (blank lines between blocks, proper table syntax). If your source has unusual formatting, it may not round-trip perfectly. You can re-run scribe export file.md after fixing the source.

For very specific journal submission templates, it's often easier to let Scribe produce clean markdown, then copy-paste into your journal's Word template.

How long does a run take?

On typical hardware with a Max subscription:

  • Write: 10-25 minutes for a 10-15k word chapter
  • Revise: 8-15 minutes for a 6-10k word document
  • Expand: 15-25 minutes for 6k → 15k words
  • Restyle: 2-5 minutes for a finished paper

Parallelism is set to 3 concurrent sections by default; you can raise it in config.yml at the cost of hitting rate limits sooner.

Is my document sent to a server?

No server that Scribe operates. Everything runs locally: the Flask UI on your machine, the Claude Code CLI spawning local processes, your files never leaving disk. API calls are made by the Claude CLI to Anthropic, billed to your own account or Max subscription.

Why does Scribe refuse to run New project on my finished document?

Because the generator treats outline.md as a structural spec, not content. If you feed it prose, it either plans around the prose headings (producing a meta- document) or, if the outline is the default scaffold, fabricates citations to fill space. We hit this in testing and it was the reason Revise and Expand exist as separate modes. Use whichever mode matches your starting point.
